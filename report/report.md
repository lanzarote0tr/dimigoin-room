# DIMIGOIN-ROOM

## 개요
현재 디미고의 인트라넷으로 사용되고 있는 소프트웨어는 DIN이라는 교내 단체에서 개발한 디미고인이라는 서비스입니다. 
디미고인은 현재 기숙사에 연관되는 일을 중점적으로 다루고 있는데, 명색이 "교내 인트라넷"이기에 학교 생활에 관한 일도 같이 적용되는 것이 좋겠다고 생각했습니다. 

그래서 저는 이번 학생회장단 공약 중 가장 중요했던 **프로젝트실 신청 절차 개편**을 실현코자 합니다. 

이 아이디어는 유튜브 체널을 만들어서 양산형 쇼츠를 올리는 것보다 더 현실성 있으며 진지한 프로젝트라고 할 수 있습니다. 
이 프로젝트를 시행함에 있어서 저 자신에게 **B2B 사업**이라는 또 다른 길을 열었다고도 생각하게 되었습니다. 

## 기획 의도
이 프로젝트는 부분적으로 온라인 프로젝트실 신청을 진행함으로써 신청자의 부담을 줄여주는 의도가 있다고 하겠습니다. 
프로젝트실을 무분별하게 신청하게 두려는 것이 아닌, 디미고답게 시간을 아낄 수 있게 하는 것에 목표가 있습니다. 

## 개발
기술스택은 현재로써는 단순 Node.js의 express와 ejs 템플릿, 데이터베이스로 mysql을 사용했습니다. 
또한 내부 filesystem과 웹서버를 분리하기 위해 docker compose를 사용하였습니다. 
디미고 이메일만이 가능한 구글 로그인을 구현하면서 구글 클라우드를 사용했습니다. 

추후에 mysql을 mariadb로 포팅하고, 프론트엔드 프레임워크를 입히는 등 서비스를 확장할 계획입니다. 

저는 개발과 deploy를 함께 했는데, 그 이유는 개발의 편의성입니다. 
서버를 두 곳에서 같이 돌리며 테스트하는 것은 불필요한 작업이 너무 많다고 생각했습니다. 

대신 `auto_deploy.sh`를 사용하여 일정 시간마다 서버가 GitHub에서 repo fetch할 수 있도록 설계했습니다. 

현재 코드는 [https://github.com/lanzarote0tr/dimigoin-room](https://github.com/lanzarote0tr/dimigoin-room)에서 확인할 수 있고, 
실제 서비스는 [https://room.trillion-won.com/](https://room.trillion-won.com/)에서 확인할 수 있습니다. 

## 인공지능 사용
저는 인공지능에게 모든 것을 맡기지 않고, 필요한 부분을 부분적으로 물어보면서 작업해 나갔습니다. 
모든 것을 인공지능에게 맡겨 개발해보니 문제 해결이 늦어지고 어려워지며 빠른 확장이 어려운 형태의 코드를 짰기에 이러한 결정을 했습니다. 

예를 들어 `auto_deploy.sh`를 서버에서 돌리기 위한 `.service` 파일 등이 필요해서:
```
Make me a automated script that lies inside the docker compose project which pulls from github every minute and recompile-reload the docker if necessary. I also need a system daemon service that runs that shell file. 
```
```
Docker Compose 프로젝트에 포함되어 1분 주기로 GitHub 변경 사항을 Pull하고, 업데이트가 있을 경우 Docker를 재빌드(Rebuild) 및 리로드하는 자동화 스크립트를 작성해 주세요. 그 쉘 파일을 실행하는 시스템 데몬 서비스도 필요합니다.
```
라는 프롬프트와 Gemini를 사용하였습니다. 

또한 css 스크립트를 수정하는 과정에서 파일과 함께:
```
center google button horizontally and vertically.
```
라는 프롬프트와 Github Copilot을 사용하였습니다. 

구글 로그인을 구현하는 중에 `passport.js`를 구현해야 한다는 사실을 알게 되어:
```
Make the google login passport config module
```
라는 프롬포트와 Github Copilot을 사용하였습니다. 

## 결과
MVP로써의 쓰임이 있는 프로그램이 단 6시간만에 탄생하였습니다. 아직 로직 구현과 디자인이 미흡하지만, 핵심 기능과 개발 환경은 구현했으므로 성공이라고 생각합니다. 

## 비즈니스모델
이 시스템은 결재 시스템으로 생각할 수 있습니다. 
단순하게는 상부에 보고하고 허락을 받는 구조라고 생각할 지 모르지만, 이러한 환경이 없는 회사가 많습니다. 
만약 이 비즈니스모델을 가지고 창업을 하게 된다면 수많은 국내 또는 해외 회사들에게 각각 입맛에 맞는 ERP를 포팅해 주는 **B2B** 사업이 될 것입니다.

이런 시스템을 조금 더 넓은 범위의 전문용어로 ERP라고 하는데, 기존의 비싼 SAP와 더존비즈온을 대체할 수 있는 신생 ERP를 만드는 기회라고 생각합니다. 
현재 ERP 시장 규모는 전 세계적으로 811억 달러로 평가된다고 합니다. 하지만 독점 구조가 심한 ERP 시장에 침투하는 것은 충분히 돈 되는 사업이 될 것이라고 생각합니다. 

## 느낀점 및 개선점
복잡하지 않은 프로젝트라면 AI를 사용하여 빠르게 MVP를 만들어낼 수 있다는 것을 몸소 느꼈습니다. 
몇년 전만 해도 불가능했던 일이 이제는 아주 빠르게 해결 가능합니다. 
심지어 제가 AI를 사용하지 않으며 했던 프로젝트보다 문제 해결 등이 5배는 빠르게 되어서 쾌적한 개발을 할 수 있었습니다. 

하지만 아직 AI가 모든 것을 하지는 못한다는 생각이 들었습니다. 
아무리 상세한 프롬포트를 주어도 쓸만한 대답은 위에서 언급한 것 처럼 아주 국소 범위에서 작은 코드를 짜 달라고 했을 때입니다. 
아주 간단한 미니미니 프로젝트를 할 때에는 좋지만, CI/CD가 필요한, 즉 context가 중요한 개발은 아직 AI의 능력 밖이라고 느꼈습니다. 

또한 Github Copilot, Gemini, ChatGPT, Perplexity 등을 같이 사용해보니 이들 각각이 갖는 장단점이 있다고 생각했습니다. 

Perplexity는 UI와 이식성이 좋다고 생각했습니다. ChatGPT는 context를 잘 읽었고, Gemini는 제 의도를 잘 파악하고 정확한 정보를 전달했습니다. Github Copilot은 코드에 바로 적용하기 쉬워서 편했습니다.
